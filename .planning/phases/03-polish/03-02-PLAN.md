Verify all SpendingEvolution files follow project conventions:

1. **CQRS Pattern Check** (App layer):
   - Query implements `IQuery<T>`
   - Handler implements `IQueryHandler<TQuery, TResult>`
   - DTOs use `required` init properties
   - No direct repository access from handlers — uses injected query interfaces

2. **Domain Layer Check** (Core):
   - No `[JsonPropertyName]`, `[BsonField]` attributes on domain classes
   - Value objects are immutable
   - No external dependencies (Avalonia, LiteDB, etc.)

3. **Infra Layer Check**:
   - Repository/query implementations use DTOs for serialization
   - LiteDB collections use entity classes, not domain classes directly
   - No UI references

4. **UI Layer Check**:
   - ViewModel inherits from `ValtModalViewModel` or `ValtViewModel`
   - Uses `ICommandDispatcher` and `IQueryDispatcher` (not direct repo access)
   - Uses `CommunityToolkit.Mvvm` source generators (`[ObservableProperty]`, `[RelayCommand]`)
   - No business logic in ViewModel (delegates to CQRS)

5. **Localization Check** (post-Plan 01):
   - Verify all hardcoded strings from Plan 01 were replaced
   - Check that menu/context menu strings use `x:Static` bindings

6. **Specific checks for SpendingEvolution**:
   - `ISpendingEvolutionQueries` is in App layer Contracts/ (not Infra)
   - `GetSpendingEvolutionHandler` uses `IQueryDispatcher` pattern correctly
   - `SpendingEvolutionQueries` (Infra) implements the contract
   - `SpendingEvolutionViewModel` disposes `CancellationTokenSource` and unsubscribes from events
   - `SpendingEvolutionChartData` properly configures LiveCharts with dual Y-axes

Document any deviations found. If deviations are minor and can be auto-fixed, fix them. If deviations require design changes, document in SUMMARY.md.
